## Verdict

The spec is a strong skeleton: the package layout is sensible, the parity-matrix idea is exactly right, the out-of-scope list is mostly honest, and the phasing is realistic. The README's product framing ("Go SDK for the ab0t mesh", drop-in promise) matches the Python lib's VISION.md intent well.

However, the spec is **not yet implementable as written**. Several contracts it pins are factually wrong against both the Python lib and the live services (invented Lua scripts, wrong HTTP methods, a nonexistent auth endpoint, an int64 counter model for a float-based system), and the config schema it specifies would **fail to load the one real production config that exists** (sandbox-platform's). One Go-specific design mechanism (idempotent-handler detection "via type assertion" on a func) is not possible in Go and needs a redesign. I also found a **live bug in the Python lib** (NameError in `setup.py`) that the spec would faithfully port.

Fix the Critical items before any code is written; the Major items before Phase 3 (billing); Minor items are doc hygiene.

## 2. Critical findings (block implementation until amended)

### C1. The Lua scripts do not exist ‚Äî the spec invents them

`PRODUCT_SPEC.md` (`counters/lua/` with `gauge_increment.lua`, `rate_check.lua`, `accumulator_check.lua`; "loaded via `redis.SCRIPT LOAD` ‚Ä¶ **same SHAs as the Python lib so behavior is bit-identical**") and `back_references.md` ("Counter impls + Lua scripts ‚Ä¶ REUSE the .lua files") describe files that are not in the Python repo. A full grep of `ab0t_quota/` finds **zero Lua, SCRIPT LOAD, or EVALSHA usage**. The Python counters are plain Redis ops:

- Gauge: `INCRBYFLOAT` on `quota:{org}:{resource}:gauge` (+ per-user partition key, + pipeline for org+user), floor-at-zero by overwrite (`counters/gauge.py`)
- Rate: sorted set `quota:{org}:{resource}:rate`, `ZADD`/`ZREMRANGEBYSCORE`/`ZCARD` (`counters/rate.py`)
- Accumulator: `INCRBYFLOAT` on `quota:{org}:{resource}:acc:{period_key}` with calendar-aligned period keys (`%Y-%m-%dT%H`, `%Y-%m-%d`, `{year}-W{week:02d}`, `%Y-%m`) (`counters/accumulator.py`)

**Action:** Rewrite the counters section to mirror these exact operations and key shapes (add them to the parity matrix ‚Äî they're currently missing, yet they are *the* shared-Redis interop surface). If you want Lua to fix Python's known check-then-act race, that's an *improvement proposal* that changes shared-Redis semantics between the two libs ‚Äî it must be called out as a deliberate divergence, not "bit-identical parity," and Python would need the same scripts to keep mixed deployments coherent.

### C2. Counter and limit types must be float, not int64

Spec defines `Counter.Check(ctx, key, limit int64) (current int64, ‚Ä¶)`, `Increment(‚Ä¶, n int64)`, and `TierLimit.Limit *int64`. The Python system is **float end-to-end**: `TierLimits.limit: Optional[float]` (`models/core.py:169`), `INCRBYFLOAT` everywhere, `ResourceDef.precision` (0‚Äì2 decimal places), and the flagship use case ‚Äî `sandbox.monthly_cost` accumulator with `precision: 2` incremented by fractional USD amounts (`record_cost` in sandbox-platform). An int64 counter cannot represent it, and an int64 limit cannot represent a `$10.50` spend cap.

**Action:** `float64` for counter values/deltas/limits (matching `INCRBYFLOAT` semantics and Redis number formatting for shared-Redis compat). Keep `config.Decimal` for *money in API bodies* only. Also note: `WithCount(n int)` should be `WithDelta(d float64)`.

### C3. The config schema as specified cannot load the real production config

`config/tier.go` et al. are missing fields that the only real consumer config (`sandbox-platform/quota-config.json`) and the Python models (`models/core.py`) actually use:

1. **`${QUOTA_*}` env interpolation** ‚Äî Python's loader expands `${VAR}` / `${VAR:-default}` in all string values, **namespaced to `QUOTA_`-prefixed vars only** (non-namespaced refs warn and resolve to the default) (`config.py:57-90`). The sandbox config uses `${QUOTA_STATE_TABLE:-ab0t_quota_state}`. Without this, the spec's own acceptance criterion ("Python deployment's JSON config loads unchanged") fails. This must be in `config/load.go`.
2. **`TierLimits` extra fields** ‚Äî `warning_threshold` (default 0.80), `critical_threshold` (default 0.95), `burst_allowance`, `per_user_limit` (`models/core.py:164-198`). The engine's ALLOW_WARNING decisions, burst soft-cap, and per-user gauges all depend on them. Limits also accept bare number / `null` / object ‚Äî spec covers that, but the object form has 5 fields, not 1.
3. **`TierConfig.billing_model` + `price`** ‚Äî 7-value `BillingModel` enum with an experimental-models gate (`AB0T_QUOTA_ALLOW_EXPERIMENTAL_BILLING_MODELS`, fail-loud at load), `Price{amount_per_period, currency, period}`, and the cross-field validators (`subscription_with_credits` requires price + invoice-paid grant; `subscription_unlock_only` forbids grant; `consumption_only` forbids price) (`models/core.py:210-602`). The sandbox config declares `billing_model` and `price` on tiers today.
4. **`initial_credit` ‚Üí `credit_grant` synthesis** ‚Äî legacy `initial_credit` auto-synthesizes a signup/persistent/credit_balance grant when `credit_grant` is unset (`models/core.py:494-503`). Spec lists `InitialCredit` but not the synthesis rule.
5. **`default_per_user_fraction` derivation** ‚Äî `ceil(limit √ó fraction)`, floored at 1.0, explicit `per_user_limit` wins (`models/core.py:466-482`).
6. **`CreditTrigger` full enum** ‚Äî 5 values (`signup`, `subscription_invoice_paid`, `scheduled_period_start`, `manual`, `webhook_admin`); `CreditGrant.rollover_max_periods` validator (required iff `rollover_capped`, forbidden otherwise); `dedup` field with 4 policies.
7. **Storage extras** ‚Äî `redis_key_prefix`, `redis_password` (and the `REDIS_PASSWORD`/`QUOTA_REDIS_PASSWORD` env convention, `setup.py:263-272`), `persistence_enabled`, `persistence_sync_interval_seconds`, `dynamodb_region`. Plus top-level `engine_mode`, `bridge_cache`, and the `pricing` section (consumed by `BudgetChecker` and the products table).
8. **Search-path divergence** ‚Äî Python: `$QUOTA_CONFIG_PATH` ‚Üí `./quota-config.json` ‚Üí `/etc/ab0t/quota-config.json` ‚Üí **built-in `DEFAULT_TIERS`** (`config.py:29-112`, `tiers.py`). Spec: `$QUOTA_CONFIG_PATH` ‚Üí `./quota-config.json` ‚Üí `./config/quota-config.json`, no defaults. Pick one deliberately; if you drop `/etc/ab0t` and the built-in default tiers, say so (recommend: keep Python's order for drop-in parity; built-in defaults are arguably a misfeature ‚Äî failing loud on missing config is fine, but then *document the deviation*).

### C4. The subscription-credit money path is missing ‚Äî configured tiers would silently never grant

The Go spec's config includes `CreditGrant` with `subscription_invoice_paid` triggers and the billing client includes `ApplyCreditGrant`/`ResetSubscriptionCredit` ‚Äî but **nothing in the spec triggers them**. In Python this is the Stripe webhook proxy (`billing/router.py` `POST /api/webhooks/stripe`) + `billing/subscription_credit.py`:

- **T1** local Stripe signature verify with `AB0T_QUOTA_STRIPE_WEBHOOK_SECRET` (Stripe SDK; multi-secret trust model documented in `docs/WEBHOOK_AND_CREDIT_GRANT_ARCHITECTURE.md` ‚Äî the 2026-05-18 prod incident was exactly this secret missing from payment's list)
- **T2** `invoice.paid` / `invoice.payment_succeeded` ‚Üí resolve org/plan from invoice metadata ‚Üí 4-level plan‚Üítier resolver (explicit `billing_integration.{plan_to_tier,stripe_price_to_tier}` maps ‚Üí plan metadata.tier_id ‚Üí price‚Üíplan‚Üímetadata ‚Üí display-name fallback) ‚Üí `apply_credit_grant` with idempotency key `invoice:{invoice_id}:credit_grant`; status matrix: transient failure ‚Üí 503 (Stripe retries), permanent ‚Üí 200 + ERROR log (no retry storm)
- **T3** `customer.subscription.updated/deleted` ‚Üí downgrade detection via `sort_order` ‚Üí `reset_subscription_credit` (idempotency: Stripe event id or `tier_change:{org}:{old}->{new}`); 409 safety-check is a non-error
- **T4** forward original body + original `Stripe-Signature` to payment-service (`forward_webhook`); plus a `checkout.session.completed` tier-sync fallback via the checkout-intent store

The spec's only nod is `payment.Client.ForwardStripeWebhook`. The lib's own design principle (the experimental-billing-model gate exists precisely so "configured behaviour actually happens" ‚Äî `models/core.py:228-233`) demands one of:

**Action (pick one, in the spec):**
- (a) Port the webhook proxy + `subscription_credit` + plan‚Üítier resolver into v0.1.0 (adds a Stripe-signature dependency ‚Äî `stripe-go` or a small manual HMAC verifier ‚Äî to the dependency table), **or**
- (b) Scope subscription billing models out of v0.1.0 and make `LoadConfig` **fail loudly** when a tier declares `billing_model: subscription_with_credits` (mirror the experimental gate). Quietly accepting the config while never granting would be the exact bug class the Python lib legislates against.

Same decision needed for the also-missing `budget.py` (`BudgetChecker`: pricing table from config, `pre_launch_check` reserve ‚Üí 402, `on_failure` refund ‚Äî exported in Python's `__all__`) and the checkout/portal/payment-methods/top-up proxy routes (Python's router mounts ~20 routes; the spec names none of them ‚Äî list them or cut them explicitly).

### C6. `authevents.Event` drops the envelope's top-level fields

Auth's delivered payload is richer than the spec's struct: top-level `tenant_id`, `user_id`, `actor_id`, `resource_id`, `resource_type`, `metadata`, `correlation_id`, `causation_id` alongside `event_type/event_id/occurred_at/data` (`appv2/events/webhook.py:243-258`). Python's dispatcher reads `data.user_id` **or top-level** `user_id` (`auth_events.py:198-199`), and falls back to `type`/`id` key aliases and a content hash (`sha256(sorted-keys JSON)[:32]`) when `event_id` is absent (`auth_events.py:196, 278-280`).

**Action:** Add the top-level fields to the Go `Event` struct (or document that extractors consult `Raw`); pin the `UserID()`/`OrgID()` fallback order (data ‚Üí top-level), the `type`/`id` aliases, and the content-hash rule in the parity matrix. Also: the receiver must accept the legacy **`X-Webhook-Signature`** header (`auth_events.py:149`) and **bare-hex** signatures without the `sha256=` prefix ‚Äî the Python CLI's own `replay`/`backfill` sign that way (`__main__.py:208-212`). Verify HMAC over the **raw received body bytes** (never re-canonicalize JSON in Go; auth signs compact sorted-keys JSON and sends those exact bytes).

### C7. "Idempotent detection via type assertion" is not implementable in Go

`handlerledger/decorator.go` says the receiver detects wrapped handlers "via a type assertion (the returned HandlerFunc carries an internal struct instead of being a bare closure)." A Go `func(ctx, Event) error` value **cannot** be type-asserted to a struct, and a named func type carries no data. Python does this with function attributes (`fn._ab0t_idempotent`, `handler_ledger.py:697-715`), which Go has no analogue for.

**Action ‚Äî redesign; viable options:**
1. **Side registry** keyed by `reflect.ValueOf(fn).Pointer()` ‚Äî `Idempotent` records its config in a package map; the dispatcher looks it up. Closest to Python; fragile with method values.
2. **Make registration interface-based**: `type Handler interface { Handle(ctx, Event) error }` with `HandlerFunc` adapter; `Idempotent` returns a concrete `*idempotentHandler` struct that the dispatcher type-switches on. Idiomatic Go; recommended.
3. Registry stores a small struct `{fn HandlerFunc, idem *IdempotentConfig}` and `OnAuthEvent` has an `‚Ä¶Opts` variant.

Option 2 keeps `OnAuthEvent(et, fn)` ergonomic (accept `HandlerFunc` and wrap) while making wrapped handlers detectable. Amend the spec with the chosen mechanism ‚Äî this shapes the whole `authevents`/`handlerledger` boundary.

### C8. Live Python bug found during this audit ‚Äî do not port it; report it

`ab0t_quota/setup.py:938` references `tier_provider=provider` inside `_wire_paid_tier_sync`, but `provider` is a local of `setup_quota` and **is not in scope** (confirmed by AST analysis: no param, no local assignment, single Load ref at line 938). Result: when `AB0T_AUTH_WEBHOOK_SECRET` is set, building the **default signup-credit handler always raises `NameError`**, swallowed by the `except` at `setup.py:967` ("auth-event webhook setup failed: name 'provider' is not defined"). The webhook receiver still mounts (line 911, before the failure), so consumers with their *own* handlers (sandbox-platform) are unaffected ‚Äî but the v0.5.2 "zero-code signup credit" promise (T11) is broken for any consumer relying on the default handler.

**Actions:** (1) file a ticket against the Python lib (one-line fix: pass `provider` into `_wire_paid_tier_sync`); (2) the Go spec's `BuildDefaultCreditGrantHandler` wiring is the same design ‚Äî spec should note the tier-provider must be threaded into the paid-tier wiring explicitly (the Go structure makes this natural; just don't copy the Python call-graph blindly); (3) add a conformance test: "default handler registers and fires when only config + env are set" ‚Äî the test Python evidently lacks.

## 3. Major findings (fix before the affected phase)

### M1. `EnablePaid` default diverges from Python ‚Äî fine, but say so
Python: `enable_paid=True` default (`setup.py:187`). Go spec: "default false". Defaulting to false is arguably safer for a v0.1.0, but the spec's own rule is "deviations must amend this file" ‚Äî record it as a deliberate deviation, and make the README example pass `EnablePaid: true` where it shows billing routes mounting.

### M2. Tier-catalog publish omitted
On startup (paid mode), Python PUTs the full quota catalog (tiers + resources + bundles, deliberately omitting billing-policy fields ‚Äî a **locked decision** with a regression test, `setup.py:606-732`) to `PUT /billing/tier-catalog/{service_name}`. Admin views (`GET /billing/{org}/tier/limits?service=‚Ä¶`) and bridge mode depend on it. The Go spec neither includes nor excludes it. Decide; if deferred with bridge mode, note that `MeshTierProvider` still works without it but admin views won't show Go consumers' limits. If ported, port the **omit-billing-policy rule and its test** too.

### M3. `LifecycleEmitter` is primarily an SNS publisher
Python's emitter publishes lifecycle events to SNS (`AB0T_MESH_SNS_LIFECYCLE_TOPIC_ARN`) and *additionally* auto-records cost on `resource.stopped/deleted` with idempotency key `cost:lifecycle:{resource_id}` (`billing/lifecycle.py`). The Go spec describes only the cost-record half and lists no SNS dependency. Decide: port SNS publish (add `aws-sdk-go-v2/service/sns` to the dependency table) or scope the Go emitter to cost-record-only and document that lifecycle fan-out is out of scope. Same for `HeartbeatMonitor` specifics worth pinning: Redis hash `heartbeat:{resource_id}` (TTL 1800s), stale threshold 900s, scan interval 60s, synthetic stop with `reason="heartbeat_timeout"`.

### M4. Engine surface is thinner than Python's in load-bearing ways
- `QuotaResult` needs the full decision model: `decision ‚àà {allow, allow_warning, deny, unlimited}`, `severity`, human `message`, `denied_level ‚àà {org, user}`, `user_current`/`user_limit`, `retry_after` (rate counters compute seconds-until-slot on denial), `utilization`, `upgrade_url`, `has_override` (`engine.py:103-198, 420-498`; `models/responses.py`). These power 429 bodies, headers, and the message builder. The spec's `QuotaResult{Denied ‚Ä¶}` sketch loses the warning band and burst behavior entirely.
- **Burst allowance**: over-limit but within `limit + burst_allowance` ‚Üí ALLOW_WARNING, not deny (`engine.py:454-463`).
- **Per-user gauge sub-quota**: only for gauges, only when `user_id` present, using `derive_per_user_limit` (`engine.py:140-181`).
- **Bundle semantics**: unknown bundle ‚Üí trivially allowed (fail-open by design); `IncrementBundle` namespaces idempotency keys as `{key}:{resource_key}`; `DecrementBundle` silently skips non-gauge members and swallows per-resource errors (`engine.py:267-345`). Pin all three.
- **`BatchCheck` overpromises**: spec says "one Redis MULTI/EXEC"; Python is a sequential loop of independent checks (`engine.py:200-221`), and `check` itself is read-then-evaluate (no atomic reserve). If Go implements true atomicity it *diverges* (and a Go-allowed/Python-denied race difference on shared Redis is observable). State the intended semantics honestly.
- Missing from the Go API: `Reset` (admin force-set with audit log), `InvalidateTierCache` (webhooks call it after tier change), `CheckFeature` exists ‚úì.

### M5. Tier caching: Redis-shared in Python, in-memory in the Go spec
`AuthServiceTierProvider` caches in **Redis** (`quota:tier:{org_id}`, TTL from config) with an `invalidate(org_id)` method (`providers.py:45-89`) ‚Äî shared across processes/replicas and invalidated by checkout-completion webhooks. The spec's `Cached(p, ttl)` in-memory wrapper breaks cross-replica invalidation. Mirror the Redis cache (same key, so Python and Go deployments share it) and expose `Invalidate`. Also `StaticTierProvider` takes a `mapping` + default in Python ‚Äî spec says "always returns free".

### M6. Receiver/dispatch contract details to pin in the parity matrix
From `auth_events.py:128-280`: 200 `{"status":"ignored","event_type":‚Ä¶}` when no handlers; multiple handlers per event type all run, each isolated; idempotent path: delivery-dedup short-circuit logs and returns; key-fn exceptions degrade to no-business-dedup with a warning; retries are **inline and blocking** within the webhook request (sleep between attempts ‚Äî note: 3 attempts √ó backoff can approach auth's 30s delivery timeout; consider documenting that auth's own retry policy plus FAILED_PERMANENT covers the overflow); terminal failure status is `failed_permanent` (CLI `replay` resets the row to `failed` to make it retryable). Also pin: handler returning nothing = SUCCESS without side_effect_id.

### M7. Grant-flow specifics the spec must pin (money correctness)
From `auth_events.py:490-630`: billing **HTTP 400 is treated like success** for flag-setting purposes (deliberate: caps/duplicates shouldn't retry forever); Redis dedup flag TTL = 30 days (`ex=86400*30`); idempotency-key shapes per all 4 dedup policies; legacy path ‚Üí `/promotional-credit`, new-schema path ‚Üí `/apply-credit-grant`; `rollover_max = rollover_max_periods √ó amount_per_period`; tier-lookup failure aborts silently (warning only); Redis-flag read failure degrades to billing-side idempotency. Each of these is observable behavior a Go consumer's money depends on.

### M8. CLI parity gaps
- `back_references` env list omits `AB0T_MESH_API_KEY` (the CLI's `subscribe-events` accepts it as fallback auth via `X-API-Key`, `__main__.py:35-49`).
- `subscribe-events` hardcodes `event_types=["auth.user.registered","auth.user.login"]` ‚Äî Go CLI must match (this differs from the lib's runtime auto-subscribe which uses registered types).
- `events` requires `--user-id` OR `--status` OR (`--handler`+`--event-id`), exit 2 otherwise; `--since` accepts `Nh/Nd/Nm/ISO`.
- `backfill` synthesizes `{"event_type", "event_id": "backfill_{uid}_{ts}", "data": {"user_id", "org_id", "_synthetic": true}}`, `--handler` is logging-only; both `replay` and `backfill` append `/api/quotas/_webhooks/auth` to a bare public URL and sign with **bare hex**.
- Note for the dependencies table: Python's CLI is argparse with `python -m ab0t_quota`; cobra is fine, but "same exit codes" should be added to the acceptance criteria (0/1/2 semantics above).

### M9. `messages.go` must not port the hardcoded copy
`messages.py` ships sandbox/auth-specific `ACTION_HINTS` and a hardcoded free‚Üístarter‚Üípro `UPGRADE_TIER_MAP`, both tagged `TODO(public-mesh-ga): move into consumer config` (`messages.py:20-52`). A new SDK aimed at external Go consumers should make hints/upgrade-targets **config-driven from day 1** (with generic fallbacks), not inherit the TODO. Cheap now, breaking later.

### M10. Dead config knobs: `enforcement.shadow_mode`, `enforcement.global_kill_switch`
Present in the example config and the real sandbox config, referenced **nowhere** in Python lib code (grep confirms; only `enforcement.enabled` is read, and only to log a warning ‚Äî it doesn't actually disable enforcement either). Spec includes `EnforcementConfig (enabled, shadow_mode)`. Decide: implement them properly in Go (shadow mode is genuinely useful for rollout ‚Äî check-and-log-but-allow), or omit and document. Don't port dead knobs silently.

### M11. Mount-prefix vs auto-subscribe coupling
`SubscribeOnStartup` composes the endpoint as `{public_url}/api/quotas/_webhooks/auth` (hardcoded, `auth_events.py:349`), while Go's `Mount(mux, prefix)` lets the consumer pick any prefix. If a Go consumer mounts at `/v1`, auto-subscribe registers a dead URL. Either fix the coupling (subscribe uses the actual mount prefix) or constrain `Mount` to document that the webhook path is fixed for mesh-parity reasons.

### M12. Per-upstream mesh credentials missing from `quota.Config`
Python supports `AB0T_MESH_BILLING_API_KEY` / `AB0T_MESH_PAYMENT_API_KEY` with fallback to the unified `AB0T_MESH_API_KEY` (`setup.py:816-818`) ‚Äî and today's mesh actually issues **separate keys per upstream provider** (per the mesh-service-accounts model: one consumer sub-org + key per provider). The Go `Config` has only `MeshAPIKey`. Add `BillingAPIKey`/`PaymentAPIKey` overrides + the env fallbacks. Also port the `_api_key_headers` heuristic: key containing `.` ‚Üí `Authorization: Bearer`, else `X-API-Key` (`clients.py:81-89`).

### M13. Auth-dependency story for the proxy routes is underspecified
Python leans on the external `ab0t_auth` wrapper for JWT validation; consumers pass `paid_auth_reader`/`paid_auth_admin` deps, and the router **fails loudly** (`ValueError`) if a reader is given without an admin (permission-boundary-collapse guard from the 20260428 audit; `billing/auth_helpers.py`). There is no `ab0t-auth-go`. The spec's `mesh/auth.go ‚Äî JWT extractor from request` quietly absorbs an entire JWT/JWKS/audience-validation subproject. Recommend for v0.1.0: the Go router takes consumer-supplied `func(http.Handler) http.Handler` middlewares for reader/admin (mirroring Python's "bring your own dep"), ports the strict-mode error verbatim, and JWKS validation ships only as an optional helper (`GET /organizations/{org_id}/.well-known/jwks.json` is confirmed public). Mesh security rules to restate in the spec verbatim from the gateway pattern: org_id comes from the validated JWT, never from request params; the consumer API key never reaches the browser; upstream error details are logged server-side and masked in responses (the Python clients raise upstream `detail` ‚Äî the **router** is where masking must happen; make this explicit in `router.go`).

### M14. DDB specifics for the ledger and quota store
Pin in the parity matrix (currently absent): GSI names are literally `gsi1`/`gsi2` with attributes `gsi1_pk/gsi1_sk/gsi2_pk/gsi2_sk` (`handler_ledger.py:536-564`); DDB TTL attribute named `ttl`; Redis `mark_done` has **no TTL** (credit dedup is permanent ‚Äî comment: "promotional credits don't expire"); quota-state schema `PK=ORG#{org_id}` with `SK ‚àà {TIER, OVERRIDE#{rk}, COUNTER#{rk}, INCREASE#{rid}}` (`persistence.py:1-15`). Two more persistence behaviors: Python **auto-creates the DDB table** on init ‚Äî recommend Go does *not* (runtime IAM should not carry CreateTable; provision via ops tooling) and documents the table spec instead; and port the SSRF allowlist for `DYNAMODB_ENDPOINT` overrides (`persistence.py:36-63`).

### M15. Middleware details
`QuotaGuard` has `exempt_paths` (default health/metrics/docs set), `enabled`, `fail_open`, `fail_open_error_threshold`, and a default org-extractor reading `request.state.user.org_id` (`middleware.py`). Go needs an explicit **context-key contract** for "authenticated org" (e.g. `quota.OrgFromContext`) since there's no `request.state`. Spec's `middleware/guard.go` line mentions none of this.

## 5. Client-DX assessment ("ensure lib is easy to use for clients")

The README's pitch is right, and the one-call `Setup` + `Mount` + `CheckBundle` shape is genuinely good. But the *real* consumer integration (sandbox-platform's `app/quota.py`, ~430 lines of wrapper) shows where the Python lib's drop-in promise leaks, and the Go spec currently re-creates most of the leaks:

1. **"Two env vars" is only true for quota-only mode.** Full paid + events mode needs ~8: `AB0T_MESH_API_KEY`, `AB0T_CONSUMER_ORG_ID`, `AB0T_AUTH_WEBHOOK_SECRET`, `AB0T_AUTH_ADMIN_TOKEN`, `AB0T_AUTH_WEBHOOK_PUBLIC_URL`, `AB0T_AUTH_AUTH_URL`, `AB0T_QUOTA_STRIPE_WEBHOOK_SECRET`, plus storage vars. Python fails *quietly* per-feature when one is missing (each subsystem logs-and-skips, scattered across startup logs). **Recommendation:** Go's `Setup` should compute a **capability report** ‚Äî `{QuotaAPI: on, RateLimit: on, Paid: off (missing AB0T_CONSUMER_ORG_ID), AuthEvents: off (missing AB0T_AUTH_WEBHOOK_SECRET)}` ‚Äî log it as one block, and expose it on `QuotaContext` (e.g. `qctx.Capabilities()`). This converts the #1 Python operability complaint ("why didn't credits grant?" ‚Äî cf. C8, which hid for a full release behind exactly this pattern) into a 5-second diagnosis. Document the per-feature env matrix in the README as a table.
2. **Expose the Redis handle.** Sandbox-platform reaches into `_ctx.engine.redis` / `_ctx._redis` defensively because review item M-3 ("public redis accessor") was never implemented in Python. Go spec: add `QuotaContext.Redis()` (or accept that consumers bring their own client ‚Äî they already pass it in `Config.Redis`, so just document that pattern).
3. **Fail-open/fail-closed asymmetry must be documented as API.** The consumer wraps pre-flight checks (fail-closed, 503 before engine-ready, with a `QUOTA_FAIL_OPEN_DURING_STARTUP` escape hatch) vs lifecycle hooks (fail-open, log-only). Go can do better: `Setup` returns only after the engine is usable (no FastAPI-lifespan ordering problem), which removes the 503 window entirely ‚Äî say so in the spec, it's a genuine Go advantage.
4. **Typed errors over status codes.** `Check` returning `*QuotaError` is right; make sure it carries `Decision`, `RetryAfter`, `Message`, `UpgradeURL`, and that `errors.As` works, and ship a ready-made `http.Handler` 429-writer (`middleware/headers.go` should include a `WriteDenial(w, err)` helper) so consumers don't hand-roll the response body. Python consumers get this via raised HTTPException; Go consumers shouldn't have to reconstruct it.
5. **Bundle dispatch hint**: consumers hardcode bundle names per route and duplicate GPU-detection logic. Cheap win: document the `WithInstanceType` option's contract (the spec lists it but never says what it does ‚Äî in Python the *consumer* maps instance type ‚Üí bundle; either implement mapping in the lib via config or drop the option).
6. **Examples directory is the right move** ‚Äî add a fifth example: `with_stripe_webhooks/` (or remove subscription billing from v0.1.0 per C4). Each example should be runnable against `internal/testutil` fakes so `go test ./examples/...` actually exercises them beyond `go build`.
7. **Global handler registry**: package-level mirrors Python by design (fine), but offer `authevents.NewRegistry()` for tests/multi-tenant binaries and make the package-level functions delegate to a default instance. `ClearHandlers` as "test-only, don't call in production" is a smell Go can fix structurally.
8. **Godoc-first**: the spec's `doc.go` per package is right; add the acceptance criterion "every exported identifier has a doc comment; `pkgsite` renders the quickstart" since external Go consumers will meet the lib through godoc, not the README.

## 6. Mesh-network alignment (per `mesh-service-accounts`)

The spec mostly matches the mesh model, with the gaps already noted (M12 per-provider keys, M13 JWT/audience story). For completeness, the Go README/spec should state the consumer-onboarding prerequisites explicitly, because "two env vars" presumes they exist:

1. The Go service is registered as a **consumer** of billing and payment (consumer sub-org under each provider org, service account, API key with `billing.read.*`/`billing.write.*` + `billing.cross_tenant`, `payment.read.*` + `payment.cross_org` ‚Äî created via `register-as-client.sh` / the setup CLI, not by this library).
2. The library's calls are exactly the **gateway pattern**: user JWT (consumer's audience) validated locally ‚Üí org_id extracted from claims ‚Üí upstream called with the consumer's `X-API-Key` ‚Üí upstream checks cross-tenant permission. The library must never forward the user's JWT upstream (wrong audience ‚Äî it would 401) ‚Äî worth an explicit test: proxy routes strip `Authorization` before upstream calls.
3. Permission names the proxies depend on (reader: `billing.read`,`costs.read`; admin: `billing.admin`,`costs.admin` ‚Äî `billing/auth_helpers.py`) should be listed in the Go README so operators can scope keys correctly.
4. Auto-subscribe writes need `events.subscribe`/`events.read` on the **auth admin token** ‚Äî a different credential than the mesh API key; the docs conflate them today.

## 8. Suggested spec amendments to the implementation order
